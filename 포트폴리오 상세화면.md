4. **기술적 결정** — A 대신 B를 선택한 이유 (트레이드오프)

## 샘플: 삼성전자 슈퍼컴퓨터 모니터링 시스템 (SMS)

### 배경/문제

삼성전자 연구소에서 운영하는 슈퍼컴퓨터 클러스터의 상태를 실시간으로 파악할 방법이 없었다.
장애 발생 시 담당자가 직접 서버실에 가서 확인하거나, 수동으로 로그를 뒤져야 했기 때문에
장애 인지까지 평균 30분 이상 소요되었고, 그 사이 연산 작업이 중단되는 손실이 반복되었다.

### 나의 역할

- **팀 규모**: 4명 (백엔드 2, 프론트엔드 1, PM 1)
- **담당 파트**: 프론트엔드 전담 — 실시간 대시보드 UI 설계 및 개발
- **추가 역할**: 백엔드 팀과 WebSocket 통신 프로토콜 설계 협업

### 기술 스택

`React` `TypeScript` `D3.js` `WebSocket` `Recharts` `Styled-Components`

### 기술적 결정

| 선택 | 대안 | 선택 이유 |
|------|------|-----------|
| **WebSocket** | Polling (REST) | 수백 개 노드의 상태를 1초 간격으로 갱신해야 했기 때문에, HTTP 폴링은 서버 부하가 과도했다 |
| **D3.js** | Chart.js | 클러스터 토폴로지를 커스텀 시각화해야 했는데, Chart.js로는 노드 맵 형태의 레이아웃을 구현할 수 없었다 |
| **Canvas 렌더링** | SVG | 노드 500개 이상일 때 SVG는 DOM 요소가 과다해져 렌더링이 느려졌다. Canvas로 전환 후 60fps 유지 |

### 어려웠던 점 & 해결

**문제**: 노드 500개 이상의 실시간 데이터를 1초마다 갱신하면 브라우저가 버벅거리는 현상 발생

**원인 분석**: 매 업데이트마다 전체 차트를 리렌더링하고 있었음

**해결**:
- 변경된 노드만 부분 업데이트하는 diff 로직 구현
- `requestAnimationFrame`으로 렌더링 주기를 브라우저 프레임에 맞춤
- Web Worker에서 데이터 가공 처리하여 메인 스레드 부하 분리

→ 렌더링 지연 800ms → 16ms로 단축

### 결과

- 장애 인지 시간: **평균 30분 → 실시간 (5초 이내)** 단축
- 슈퍼컴퓨터 노드 **500개 이상** 동시 모니터링
- 삼성전자 연구소 관제실 **대형 디스플레이**에 실제 배치 운영
- 프로젝트 완료 후 타 부서에서도 도입 요청

### 스크린샷/다이어그램

> (여기에 실제 이미지 추가)
> - 대시보드 메인 화면
> - 클러스터 노드 맵 시각화
> - 장애 알림 화면
> - 시스템 아키텍처 다이어그램

### 배운 점 / 회고

- **대량 실시간 데이터 렌더링**은 "얼마나 안 그리느냐"가 핵심이라는 것을 체감했다.
  전체를 다시 그리는 것보다 변경분만 추적하는 것이 성능에 결정적이었다.
- 다시 한다면 처음부터 **Canvas 기반 설계**로 시작했을 것이다.
  SVG로 시작해서 중간에 전환하느라 2주를 추가로 소요했다.
- 백엔드와 **WebSocket 메시지 포맷을 먼저 합의**한 것이 개발 속도에 크게 기여했다.
  API 설계를 나중에 맞추려 했다면 양쪽 다 재작업이 필요했을 것이다.
